# VistaCNC P2-S pendant protocol map

This document records the observed USB HID protocol for the VistaCNC P2-S pendant with USB vendor ID `04d8` and product ID `fce2`. Current decoding decisions are based on fresh captures from the physical pendant; historical driver variable names are not treated as protocol evidence.

## Transport

- Vendor-defined USB HID device.
- Input interrupt endpoint in the old libusb driver: `0x81`.
- Output interrupt endpoint: `0x01`.
- Input payload: 8 bytes (a platform HID API may expose a leading report-ID byte).
- Display output payload: 20 bytes. Bytes 0-15 are the visible space-padded display text, byte 16 is non-visible/unknown, byte 17 is an incrementing sequence number, and bytes 18-19 are zero in the old driver.
- The HID descriptor declares 9-byte input and 21-byte output reports at the Windows API boundary; each includes a leading report ID of zero before the USB payload.

## Input report

| Byte | Bits | Meaning |
| --- | --- | --- |
| 0 | 0-7 | Wheel position, wrapping unsigned counter. Delta is `(int8_t)(current - previous)`. |
| 1 | 0-7 | Unsigned wheel-rate code, updated about every 50 ms. It is five times the wheel counts observed in the sampling window, so `code * 4` gives counts/second. Direction is supplied separately by byte 0 deltas. |
| 2 | 0-2 | Right selector. |
| 2 | 3 | Unknown. |
| 2 | 4 | Pendant/machine-off indication. |
| 2 | 5 | E-stop level: 1 while physically latched, 0 after reset. No separate reset pulse was observed. |
| 2 | 6-7 | Wheel push-button state: 0 released, 1 ordinary press, and 3 a firmware-recognized second press of a double-click. Value 2 has not been observed. |
| 3 | 0-2 | Left selector. |
| 3 | 4 | Periodic device heartbeat. It alternates independently of controls, usually after roughly 0.5-0.7 seconds. |
| 3 | 3, 5-7 | Unknown. |
| 4-5 | 0-15 | Big-endian unsigned firmware motion accumulator. One isolated detent starts it at `0x0222` (546), after which it decays to zero over roughly 0.5 seconds. Sustained motion raises it, but fast rotation makes it wrap at 16 bits. It is diagnostic and must not directly command velocity. |
| 6-7 | 0-7 | Unknown. |

Left selector values confirmed by physical capture: 1 Step, 2 Velocity, 3 Continuous, 4 Speed, 5 Zero, and 6 Function. The physical Off detent continues to report value 6 and sets byte 2 bit 4; value 7 was not observed. The old driver's synthesized `OFF` state should not be mistaken for a raw selector value.

Right selector values confirmed by physical capture: 1 X/F1, 2 Y/F2, 3 Z/F3, 4 Start/Pause, 5 Stop/Back, 6 Spindle, and 7 A/F4. Some transitions around A/F4, Start/Pause, and Stop/Back briefly reported 0 for about one 100 ms input-report interval. Value 0 must be treated as a break-before-make/transient selector state, not immediately interpreted as a command.

## Wheel push-button capture

Physical capture confirms these report sequences:

```text
single click:             0 -> 1 -> 0
recognized double-click: 0 -> 1 -> 0 -> 3 -> 0
unrecognized click pair: 0 -> 1 -> 0 -> 1 -> 0
press and hold:           0 -> 1 ... 1 -> 0
```

Value 3 is generated by the pendant firmware on the second press only when its internal timing rule recognizes a double-click. A varied-speed capture produced three value-3 reports among 25 press reports. Ordinary second presses remained value 1. The exact firmware timing boundary is not yet established, and click history may carry between test attempts unless they are separated by a sufficiently long idle interval.

## Wheel capture

Byte 0 is an unsigned wrapping position counter. Each captured detent changed it by exactly one; signed delta must be calculated with wrapping 8-bit subtraction. Rotation while the push-button was held did not change that encoding. A held three-detent capture produced three `-1` changes while button state remained 1. Direction should be described as positive/negative until clockwise orientation is explicitly reconfirmed.

Byte 1 was decoded with released/held-button captures in both directions and at different speeds. An isolated count produced rate code 5. During sustained rotation, the code remained a multiple of 5 and matched five times the number of byte-0 counts in the approximately 50 ms firmware window. Slow and fast runs averaged about 110 and 452-490 counts/second respectively; their rate-code maxima were 45 and 145-165. Holding the wheel button did not change the encoding.

Bytes 4-5 are not required to recover direction or rate. The isolated-detent decay sequence was `0x0222`, `0x01ec`, `0x01b3`, `0x017d`, `0x0147`, `0x0111`, `0x00db`, `0x00a5`, `0x006c`, `0x0036`, and zero at roughly 50 ms intervals. Fast rotation reached the 16-bit boundary and wrapped, so this field cannot safely be interpreted as an absolute speed command.

NGC Velocity mode therefore timestamps signed byte-0 detents at receipt rather than using the quantized byte-1 rate code. Motion requires a fresh ordinary wheel-button press and two same-direction detent reports close enough to measure an interval. Later detents update a smoothed interval-derived velocity. Byte 4-5 accumulator changes provide periodic opportunities to renew the command or detect a missed-detent deadline, but their magnitude does not determine or prolong velocity. Direction reversal resets the estimator and requires a new pair; button release independently requests a stop, and the backend dead-man lease stops motion if reports or renewals cease.

## Capture procedure

Build and run `vistacnc_p2s_probe`. It prints a timestamp, all eight raw bytes, decoded known fields, and which byte offsets changed. Capture one action at a time, returning the control to its prior state between actions.

Use `vistacnc_p2s_probe --display "text"` to write up to 16 characters to the LCD while continuing to monitor input reports. Longer text is truncated. Windows HID and Linux hidraw both receive a leading zero report-ID byte; the old libusb driver omitted it because it wrote the 20-byte payload directly to the interrupt endpoint.

## LCD capture

The LCD is two rows of eight characters. Payload bytes 0-7 fill the top row and bytes 8-15 fill the bottom row. The test string `NGCSCREEN` displayed as `NGCSCREE` on the top row and `N` at the start of the bottom row. `12345678ABCDEFGH` filled both rows exactly. Writing an additional `X` in payload byte 16 caused no visible change.

A one-shot Windows output report was accepted by the HID API but did not update the display. Continuous `HidD_SetOutputReport` calls alone left the firmware's `LinuxCNC` text on screen. The observed working Windows sequence continuously issues both overlapped `WriteFile` and `HidD_SetOutputReport` with an incrementing byte 17. The exact minimum refresh rate and timeout have not been measured. Both calls can echo display bytes into the input stream as false pendant reports, including apparent selector and safety-level changes. Captures contained both the complete first eight display bytes and a four-byte display prefix followed by zeros. Output uses a handle separate from the outstanding input read, and the model driver discards these recognized echoes before decoding controls.

NGC publishes changed display presentation to a dedicated output worker immediately. The worker also refreshes unchanged text every 20 ms because physical testing showed that the pendant firmware responds more promptly to closely repeated output reports. Normal modes identify the active work coordinate system and selected axis on the top row, for example `G54 X`, and show signed work position on the bottom row, for example `+99.9999`. The displayed position subtracts both the active work offset and active tool offset from machine position.

Zero mode uses the layout `X+99.999` on the top row and `G54-.001` on the bottom row. The top value is the live coordinate in the active WCS. The bottom value is a staged target coordinate at the current physical point, adjusted in configured fine-distance increments while the ordinary button is held. Button release retains the staged value; a firmware-recognized double click commits it to the active WCS. Selector and safety changes discard staging.

1. Record the untouched idle report.
2. Move the left selector through every position in physical order.
3. Move the right selector through every position in physical order.
4. Turn the wheel one detent clockwise, then one detent counter-clockwise.
5. Press and hold the wheel button, then release it.
6. Single-click, then double-click at several speeds. Preserve the complete press/release report sequence and timing.
7. Exercise E-stop and the off control only when it is safe to do so.

Do not infer safety semantics from the bit labels in the old LinuxCNC driver. Integration must separately define authoritative E-stop and machine-power behavior.

## E-stop capture

Physical capture showed byte 2 bit 5 changing directly from 0 to 1 when the E-stop was latched, remaining stable while latched, and returning directly to 0 when mechanically reset. The HID report provides switch state, not independent activate/reset events. Any future safety design must not treat a userspace HID report as the sole safety authority.
