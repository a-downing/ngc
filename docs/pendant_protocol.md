# MPG P2S pendant protocol map

This document records the observed USB HID protocol for the pendant with USB vendor ID `04d8` and product ID `fce2`. The original LinuxCNC driver in `mpg_p2s` is the source for the initial map; entries marked **unverified** still need capture from the physical pendant.

## Transport

- Vendor-defined USB HID device.
- Input interrupt endpoint in the old libusb driver: `0x81`.
- Output interrupt endpoint: `0x01`.
- Input payload: 8 bytes (a platform HID API may expose a leading report-ID byte).
- Display output payload: 20 bytes. Bytes 0-15 are the visible space-padded display text, byte 16 is non-visible/unknown, byte 17 is an incrementing sequence number, and bytes 18-19 are zero in the old driver.
- The HID descriptor declares 9-byte input and 21-byte output reports at the Windows API boundary; each includes a leading report ID of zero before the USB payload.

## Input report

| Byte | Bits | Meaning |
| --- | --- | --- |
| 0 | 0-7 | Wheel position, wrapping unsigned counter. Delta is `(int8_t)(current - previous)`. |
| 1 | 0-7 | Wheel-motion auxiliary field. Held-wheel capture showed a brief value 5 at detent activity and 0 otherwise; exact meaning is unknown. |
| 2 | 0-2 | Right selector. |
| 2 | 3 | Unknown. |
| 2 | 4 | Pendant/machine-off indication. |
| 2 | 5 | E-stop level: 1 while physically latched, 0 after reset. No separate reset pulse was observed. |
| 2 | 6-7 | Wheel push-button state: 0 released, 1 ordinary press, and 3 a firmware-recognized second press of a double-click. Value 2 has not been observed. |
| 3 | 0-2 | Left selector. |
| 3 | 4 | Periodic device heartbeat. It alternates independently of controls, usually after roughly 0.5-0.7 seconds. |
| 3 | 3, 5-7 | Unknown. |
| 4-5 | 0-15 | Wheel-motion auxiliary value. It becomes nonzero around a detent and decays toward zero over subsequent reports. Byte 4 is the high byte in the observed sequence (for example `02 22`, `01 ec`, ..., `00 36`, `00 00`). It likely represents motion timing or velocity, but its unit and authoritative meaning are unknown. |
| 6-7 | 0-7 | Unknown. |

Left selector values confirmed by physical capture: 1 Step, 2 Velocity, 3 Continuous, 4 Speed, 5 Zero, and 6 Function. The physical Off detent continues to report value 6 and sets byte 2 bit 4; value 7 was not observed. The old driver's synthesized `OFF` state should not be mistaken for a raw selector value.

Right selector values confirmed by physical capture: 1 X/F1, 2 Y/F2, 3 Z/F3, 4 Start/Pause, 5 Stop/Back, 6 Spindle, and 7 A/F4. Some transitions around A/F4, Start/Pause, and Stop/Back briefly reported 0 for about one 100 ms input-report interval. Value 0 must be treated as a break-before-make/transient selector state, not immediately interpreted as a command.

## Wheel push-button capture

Physical capture confirms these report sequences:

```text
single click:             0 -> 1 -> 0
recognized double-click: 0 -> 1 -> 0 -> 3 -> 0
unrecognized click pair: 0 -> 1 -> 0 -> 1 -> 0
press and hold:           0 -> 1 ... 1 -> 0
```

Value 3 is generated by the pendant firmware on the second press only when its internal timing rule recognizes a double-click. A varied-speed capture produced three value-3 reports among 25 press reports. Ordinary second presses remained value 1. The exact firmware timing boundary is not yet established, and click history may carry between test attempts unless they are separated by a sufficiently long idle interval.

## Wheel capture

Byte 0 is an unsigned wrapping position counter. Each captured detent changed it by exactly one; signed delta must be calculated with wrapping 8-bit subtraction. Rotation while the push-button was held did not change that encoding. A held three-detent capture produced three `-1` changes while button state remained 1. Direction should be described as positive/negative until clockwise orientation is explicitly reconfirmed.

Bytes 1, 4, and 5 also become active around wheel detents. They are not required to recover position counts, and their meaning should not be assumed from the old variable names (`counter1`, `counter2`, and `counter3`).

## Capture procedure

Build and run `pendant_probe`. It prints a timestamp, all eight raw bytes, decoded known fields, and which byte offsets changed. Capture one action at a time, returning the control to its prior state between actions.

Use `pendant_probe --display "text"` to write up to 16 characters to the LCD while continuing to monitor input reports. Longer text is truncated. Windows HID and Linux hidraw both receive a leading zero report-ID byte; the old libusb driver omitted it because it wrote the 20-byte payload directly to the interrupt endpoint.

## LCD capture

The LCD is two rows of eight characters. Payload bytes 0-7 fill the top row and bytes 8-15 fill the bottom row. The test string `NGCSCREEN` displayed as `NGCSCREE` on the top row and `N` at the start of the bottom row. `12345678ABCDEFGH` filled both rows exactly. Writing an additional `X` in payload byte 16 caused no visible change.

A one-shot Windows output report was accepted by the HID API but did not update the display. Continuously rewriting the report with an incrementing byte 17, as the old LinuxCNC driver does, updated it successfully. The exact minimum refresh rate and timeout have not been measured. Windows `WriteFile` and `HidD_SetOutputReport` were both issued during this test, so a future isolation test should determine which one is actually necessary.

1. Record the untouched idle report.
2. Move the left selector through every position in physical order.
3. Move the right selector through every position in physical order.
4. Turn the wheel one detent clockwise, then one detent counter-clockwise.
5. Press and hold the wheel button, then release it.
6. Single-click, then double-click at several speeds. Preserve the complete press/release report sequence and timing.
7. Exercise E-stop and the off control only when it is safe to do so.

Do not infer safety semantics from the bit labels in the old LinuxCNC driver. Integration must separately define authoritative E-stop and machine-power behavior.

## E-stop capture

Physical capture showed byte 2 bit 5 changing directly from 0 to 1 when the E-stop was latched, remaining stable while latched, and returning directly to 0 when mechanically reset. The HID report provides switch state, not independent activate/reset events. Any future safety design must not treat a userspace HID report as the sole safety authority.
